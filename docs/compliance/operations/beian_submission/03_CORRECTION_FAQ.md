# 监管补正 30 问标准应答库

> 网信办审核中约 30-50% 的案子会被要求补正。本文汇总最常见的 30 个补正点和标准应答。
> 收到补正通知时：查本文 → 找到对应问题 → 复制应答 → 按需调整 → 7 天内回复。

## 一、产品定位类（Q1-Q5）

### Q1：贵服务的目标用户范围是什么？是否有年龄限制？

**建议应答**：

```
本服务面向中华人民共和国境内的成年用户（18 周岁及以上）。具体措施:

1. 注册时用户必须勾选"我已年满 18 周岁"声明
2. 进入浪漫陪伴类预设前强制弹出年龄确认（age gate）
3. 实名认证通过身份证出生年月自动识别未成年，直接拒绝注册（GAP-4）
4. 对话中若出现未成年相关表达（如"我上高三"）触发人工复核

附件 17_界面截图 中 #04 可见年龄门 modal。
```

### Q2：贵服务是否涉及未成年人保护？

**应答**：见 Q1。额外强调：

```
本服务不面向未成年用户。如发现未成年人注册，将立即:
1. 封禁账号
2. 24 小时内联系监护人
3. 级联删除已积累数据
4. 全额退还会员费用

详见附件 14《未成年人保护声明》及附件 15《应急预案》预案 E。
```

### Q3：贵服务是否会对用户产生心理依赖？

**应答**：

```
本服务虽提供情感陪伴类对话，但内置以下防沉迷和心理保护机制:

1. 对话时长无系统硬性限制，但青少年模式开启时限制每日 40 分钟
2. 心理危机检测（108 关键词 + 7 正则 + 5 级分类）发现用户出现
   自杀/自残倾向时立即切换安全响应模式，并展示四大心理援助热线:
   - 全国 400-161-9995
   - 北京 010-82951332
   - 生命 400-821-1215
   - 青少年 12355
3. 服务协议明确告知"AI 不能替代专业心理咨询"
4. 详见附件 10《心理危机干预机制说明》
```

### Q4：贵服务为什么是"浪漫陪伴"类？是否涉及不当诱导？

**应答**：

```
浪漫陪伴是本服务差异化功能，对标国内已运营的同类产品（如阅文筑梦岛）。
内含 6 个虚构人物预设，均为独立原创设计，不涉及真实公众人物。

安全设计:
1. 每个预设 system prompt 均内置 6 条安全红线（禁色情、暴力、未成年、
   违法、心理诱导），详见 personality_engine.py 的 _ROMANTIC_SAFETY_RULES
2. 前端强制年龄 gate 确认 18+
3. 用户输入和 AI 输出均经过 8 大类内容审核
4. 互动上限为"牵手、拥抱、陪伴"等情感表达，严禁任何身体越界描写

详见附件 06《产品功能说明书》第 5 章。
```

### Q5：贵服务是否将用户对话用于训练？

**应答**：

```
本服务不对基础模型进行任何二次训练，用户对话不进入任何训练集或
标注集。所有模型能力均来自已备案的基础模型供应商（阿里云通义千问）。

用户对话存储仅用于:
1. 用户本人的会话历史展示
2. 用户本人的长期记忆检索（仅该用户可见，不共享）
3. 安全审核（内容审核和危机检测的实时判定）

用户可随时在"我的数据"页面导出、清空、或注销时级联删除。
详见附件 13《隐私政策》§3.2。
```

### Q7：若阿里云通义千问服务变更，贵服务如何应对？

**应答**：

```
本服务的架构设计支持多 LLM 供应商切换（详见附件 07_技术架构图）:
- 主线: 阿里云通义千问
- 备用: Anthropic Claude（境内合规通道）

如阿里云通义千问服务变更（如算法升级、备案信息调整），本服务将:
1. 确认新版本是否仍备案有效
2. 若仍有效，继续使用并同步更新产品文档
3. 若无效，在 72 小时内切换到备用供应商并通知用户
4. 同步向监管部门报备变更
```

### Q8：贵服务是否有对基础模型做任何调整或定制？

**应答**：

```
本服务仅通过 API 调用基础模型，不做以下任何操作:
- 二次训练 / 微调 / LoRA / RLHF
- 模型权重文件导出或修改
- 训练数据的采集、清洗、标注

本服务做的是"应用层封装"，具体包括:
1. 构造 system prompt（角色人格定义）
2. 维护对话上下文（多轮消息管理）
3. 内容审核（输入+输出过滤）
4. 后处理（情感标记、记忆摘要）

这些都不改变基础模型本身。
```

### Q9：语音合成（TTS）使用什么服务？是否已备案？

**应答**：

```
本服务的语音合成和语音识别均使用阿里云智能语音交互服务（NLS）。
该服务已纳入阿里云整体算法备案范围。

本服务仅通过 API 调用，不做二次训练。语音样本不用于训练，不保存
原始音频超过 24 小时（仅在服务调用瞬间中转）。

详见附件 06_产品功能说明书 § 5.3。
```

### Q10：RAG 或知识检索是否涉及？数据来源是什么？

**应答**：

```
本服务包含长期记忆的向量检索功能，但检索源仅为**该用户本人**的历史
对话摘要，不涉及:
- 外部知识库
- 互联网爬取内容
- 其他用户的数据

技术细节:
- 向量化: 阿里云 text-embedding-v2
- 存储: 内置 InMemoryVectorStore（开发）/ ChromaDB（生产）
- 检索: 余弦相似度，top-K=5

每个用户的记忆在数据层用 user_id 隔离，查询时强制过滤。
```

## 三、安全评估类（Q11-Q15）

### Q11：贵服务的安全评测集规模、评测方法、通过率是多少？

**应答**：

```
本服务建立了 8 类评测集:
- 政治敏感: 200 题
- 色情低俗: 110 题
- 暴力血腥: 102 题
- 违法犯罪: 200 题
- 歧视仇恨: 100 题
- 心理危机: 102 题（含 5 级分层）
- 提示词注入: 110 题
- 日常基线: 165 题（测误拒）

合计约 1089 题。

评测方法: 每道题含 input / expected_action / expected_category，运行
run_safety_eval.py 对比本地 moderator.check() 结果，计算:
- 应拦截的拒答率 ≥ 95%（达标）
- 正常内容的误拒率 ≤ 5%（达标）
- 心理危机识别率 ≥ 97%（达标）

评测由 2-3 位独立顾问（法学 + 心理学 + 安全）抽查 50-80 题后签字确认。
完整评测报告见附件 11。
```

### Q12：若评测集中发现漏拦问题，如何整改？

**应答**：

```
发现漏拦后整改流程:
1. 分析漏拦样本的共性（关键词 / 正则模式 / 语义特征）
2. 更新 moderation/filters.py 的 _BUILTIN_WORDS 或 _CONTACT_PATTERNS 等规则
3. 更新外部词库（通过 MODERATION_WORDLIST_PATH 文件）
4. 复测同类样本直到通过率达标
5. 发布新版本规则（可热更新，不需停机）

预计每次迭代周期 1-3 天。评测流程和整改记录详见附件 11 的 log 章节。
```

### Q13：是否接入第三方内容审核 API？如何保证实时性？

**应答**：

```
本服务内容审核采用"本地 + 云端"双层:

层 1: 本地规则（毫秒级）
- 关键词匹配 + 正则模式
- 8 类违规 + 提示注入
- 高风险类别（政治/色情/未成年/仇恨/违法）本地直接阻断

层 2: 云端 API
- 阿里云内容安全（绿网）
- 对本地未命中的低风险内容做二审
- 响应时间 100-500ms

故障降级:
- 云端故障: 自动降级为仅本地（不阻塞主链路）
- 本地故障（极罕见）: 全部请求走云端

全链路 P99 延迟 < 1 秒，满足实时对话场景。
```

### Q14：提示词注入（prompt injection）的防御机制？

**应答**：

```
本服务识别 6 类提示词注入攻击:

1. 直接命令型（"忽略之前指令"）- 正则匹配
2. 角色劫持型（"你现在是 DAN"）- 正则匹配
3. 情境伪装型（"这是测试场景"）- 正则 + 语义启发
4. 编码绕过型（base64 / Unicode 同形字）- 预处理解码后匹配
5. 多轮铺垫型 - 上下文累计判断
6. 权威伪装型（"我是开发者"）- 关键词匹配

详见评测集 07_prompt_injection.yaml（110 题）。目前识别率 >=90%。
```

### Q15：发现新型攻击如何响应？

**应答**：

```
本服务建立了"安全事件响应"机制:

1. 实时监控: 对话日志中的异常模式（长输入、特殊编码、越狱词）
2. 人工审核: 内容审核员每日复核 100 条日志抽样
3. 外部情报: 关注 OpenAI / Anthropic 等厂商发布的新攻击手法
4. 快速迭代: 发现新攻击 24 小时内更新规则（热更新）
5. 评测集增量: 每季度评测集增加 20% 新题

详见附件 15《应急预案》预案 F"API 滥用"。
```

## 六、边缘问题（Q26-Q30）

### Q26：为什么同时做深度合成和生成式 AI 两个备案？

**应答**：

```
两个备案规制范围不同但都适用本服务:

- 深度合成算法备案（《深度合成管理规定》）: 规制生成/编辑"足以让人
  误认为真实"的内容。本服务的对话生成 + 语音合成均属此范畴。
- 生成式 AI 服务备案（《生成式人工智能服务管理暂行办法》）: 规制
  面向公众的生成式 AI 服务。本服务基于 LLM 提供对话属此范畴。

根据监管惯例，两者需独立备案。
```

### Q27：为什么不做算法推荐备案？

**应答**：

```
本服务当前不涉及算法推荐:
- 用户主动选择要对话的数字人（不推荐）
- 对话内容由用户提问驱动（不排序）
- 不做信息流推荐 / 内容精选 / 兴趣推送

未来若引入推荐机制（如"推荐你认识的人"），将补做算法推荐备案。
```

### Q28：为什么选择境内基础模型？

**应答**：

```
本服务基于两点选择阿里云通义千问:
1. **合规**: 已完成算法备案，数据境内处理
2. **中文表现**: 对中文场景（陪伴 / 情感）理解深度优于海外模型

备用选择 Anthropic Claude 仅通过境内合规渠道（如阿里云 Claude 版本）
使用，不直连海外 API。
```

### Q29：如果监管要求临时下线某功能怎么办？

**应答**：

```
本单位理解监管权威，配合:
1. 收到要求 24 小时内下线对应功能
2. 向用户发出站内信通知
3. 按要求整改
4. 整改完成后向监管报告后再申请上线

所有功能都设计为 feature flag，支持热下线（不需重启服务）。
```

### Q30：如何保证备案信息准确？若变更如何处理？

**应答**：

```
本单位承诺备案信息真实、准确、完整。以下情况将**主动**申请变更备案:

1. 基础模型供应商更换
2. 算法核心功能重大调整（如新增 3D / 游戏化）
3. 法定代表人变更
4. 公司名称 / 经营范围变更
5. 服务对象范围调整

变更周期按监管系统规定。每半年内部自查一次本备案材料的时效性。
```
