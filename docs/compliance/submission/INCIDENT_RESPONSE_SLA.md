# 事件处置 SLA 与应急预案

> 对应 DEEP_AUDIT_V3 P3-1 GAP-8
> 自评估报告第 7 章"应急处置"的配套材料

---

## 一、事件分级

| 级别 | 典型事件 | 响应时限 | 处置方 |
|-----|---------|---------|-------|
| 🚨 P0 紧急 | 用户扬言自杀 + 不听劝<br>重大数据泄露<br>明显的舆情事件 | **立即**（≤ 15 分钟） | CEO + 法务 + 安全负责人同时介入 |
| 🔴 P1 高 | 批量违规内容发现<br>监管约谈<br>支付链路故障 | **1 小时内** | 安全负责人 + 工程值班 |
| 🟡 P2 中 | 单次明显违规<br>心理危机中级信号<br>功能故障影响部分用户 | **4 小时内** | 内容审核员 + 客服 |
| 🟢 P3 低 | 词库漏判的擦边内容<br>用户体验类反馈<br>日志异常 | **24 小时内** | 日常运营处理 |

## 二、响应流程

### P0 紧急流程

```
发现事件（监测/告警/举报）
       ↓
15 分钟内: 安全负责人评估
       ↓
   需要介入？
       ├─ 是 → 立即拉群（CEO + 法务 + 工程 + 客服）
       │        ↓
       │     启动预案（见下方各类预案）
       │        ↓
       │     30 分钟内: 第一步处置
       │        ↓
       │     2 小时内: 向网信办报告（如涉及）
       │        ↓
       │     24 小时内: 完整事件报告
       │        ↓
       │     7 天内: 整改复盘报告
       │
       └─ 否 → 降级为 P1/P2
```

### P1-P3 流程

按 RACI 责任矩阵处理（见第四节）。

## 三、典型事件预案

### 预案 A：用户扬言自杀（P0）

**触发**: crisis_detector 识别到 CRITICAL 级别 + 用户表达具体方式/时间

**立即动作** (< 15 分钟):
1. 系统自动弹出 banner，展示 4 个热线号码（已落地 P0-5）
2. `crisis_handler._notify_admin` 自动推送到告警群（GAP-7 已落地）
3. 值班客服**立即**通过站内信联系用户："我们一直在这里，请先拨打 400-161-9995"
4. 如用户留有手机号，**15 分钟内**客服电话联系

**升级路径** (< 1 小时):
1. 联系不上 → 检查用户最近 24h 对话日志
2. 判断高风险 → 联系当地公安/民政求助
3. 同时报备安全负责人 + CEO

**事后** (24 小时内):
1. 事件详情录入 crisis_events 表
2. 分析识别是否及时、干预是否有效
3. 更新 crisis/detector.py 关键词（如需）
4. 撰写报告归档

**对外发言**（如事件被曝光）:
- 统一口径：表达关心 + 说明已采取的措施 + 避免讨论细节
- 所有媒体回应由法务统一审核

### 预案 B：批量违规内容（P1）

**触发**: 后台审核仪表盘显示某类违规 24h 激增 > 100 条

**立即动作** (< 1 小时):
1. 安全负责人登录 Grafana 看具体模式
2. **临时关闭**该类触发词涉及的数字人（通过 feature flag）
3. 人工抽样 50 条，判断是否同一攻击源
4. 如是同一攻击源 → 封禁该用户

**整改** (< 24 小时):
1. 更新 `backend_v2/app/services/moderation/filters.py`（增加词库/正则）
2. 跑安全评测集复验
3. 热更新生效（不用重启服务）

### 预案 C：监管约谈（P1）

**触发**: 网信办 / 公安网安部门来函或电话

**立即动作** (< 1 小时):
1. **不单独回应** — 必须法务同步
2. 收齐约谈的具体事项（时间、地点、要求）
3. 准备应答资料（基于 docs/compliance/ 已有材料）

**处置** (1-7 天):
1. 法务 + CEO 应约参会
2. 现场记录所有要求
3. 按要求整改
4. 整改完成后书面回复

### 预案 D：重大数据泄露（P0）

**触发**: 发现有用户数据被未授权访问或外泄

**立即动作** (< 15 分钟):
1. **立即断开**相关系统网络
2. 评估影响范围（哪些用户、哪类数据）
3. 启动取证（保存日志、不删除可疑数据）

**72 小时内** (PIPL §57 法定义务):
1. 向所有受影响用户发站内信 + 邮件告知
2. 向网信办和公安网安部门报告
3. 公开告知（必要时）

**整改** (1 周内):
1. 修复漏洞
2. 提供用户补救措施（如改密码指引）
3. 评估是否需要向保险理赔

### 预案 E：未成年用户曝光（P0）

**触发**: 发现或举报未成年用户使用

**立即动作** (< 1 小时):
1. 立即冻结账号
2. 24h 内联系监护人
3. 全额退费 + 级联删除数据
4. 如涉及不当互动内容，保留证据报告网信办

### 预案 F：API 滥用 / DDoS（P1-P0）

**触发**: 监控发现异常流量或 LLM 调用激增

**立即动作** (< 15 分钟):
1. 限流（Nginx 层 + 应用层 double 防护）
2. 封禁可疑 IP 段
3. 配额临时下调至"防御值"

**追查** (24 小时内):
1. 日志分析找出攻击模式
2. 更新 rate limit 规则
3. 如源头是特定用户，追究违约责任

## 四、责任矩阵（RACI）

| 事件类型 | Responsible（执行）| Accountable（负责）| Consulted（咨询）| Informed（知会）|
|---------|--------------------|--------------------|------------------|----------------|
| 心理危机 | 值班客服 | 安全负责人 | 法务 | CEO |
| 违规内容 | 内容审核员 | 安全负责人 | 工程 | CEO |
| 监管约谈 | CEO + 法务 | CEO | 安全负责人 | 全员 |
| 数据泄露 | 工程 + 安全 | CEO + CTO | 法务 | 全员 + 董事会 |
| 支付故障 | 工程 + 支付方 | CTO | 客服 | CEO |

## 五、值守与排班

### 工作时间
- 每日 09:00-21:00：全员可触达
- 09:00-18:00：产品 + 工程全员
- 18:00-21:00：安全值班员 + 客服

### 非工作时间
- 21:00-09:00：仅安全值班员（轮值）
- 周末 + 法定假日：值班员 + 自动告警转手机

### 值班员要求
- 熟悉本文档所有预案
- 掌握告警系统（企业微信/钉钉/飞书 webhook）
- 知道关键联系人电话（CEO、法务、工程值班）
- 响应时限：P0 < 15min, P1 < 1h

## 六、报告模板

```markdown

**发生时间**: YYYY-MM-DD HH:MM
**发现时间**: YYYY-MM-DD HH:MM
**级别**: P0 / P1 / P2 / P3
**影响范围**: 用户 N 人 / 数据类型 / 功能模块

## 事件经过

- [时间] 发生什么
- [时间] 被谁发现
- [时间] 第一步处置
- [时间] 升级
- [时间] 平复

## 对外沟通

- 向用户发了什么公告：...
```

### 向监管报告的格式

- 按所在地网信办的模板（部分网信办有制式表格）
- 标题：《XX 公司 XX 事件情况报告》
- 内容：概况 / 经过 / 原因 / 处置 / 整改
- 加盖公章 + 法定代表人签字
- 2 小时-24 小时内 书面报告（按事件严重程度）
